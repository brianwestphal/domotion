# Domotion docs

Requirements / design docs for the rendering pipeline. Numbered to match the order they were authored; numbering does not imply dependency order.

> **Cross-platform note**: Domotion is an npm package and is expected to function on macOS, Linux, and Windows. Today the rendering pipeline is fully calibrated only against Chromium-on-macOS (CoreText-based fallback chain); Linux (fontconfig — Noto / DejaVu / Liberation) and Windows (DirectWrite — Arial / Consolas / Segoe UI Symbol / Cambria Math) calibration is roadmap work tracked in DM-258 (path discovery) → DM-259 (Linux) / DM-260 (Windows) / DM-261 (bundled fallback fonts) → DM-262 (per-platform CI). Treat any macOS-only code path as debt to flag, not a design choice — match Chromium's actual painted output on each target platform.

| [70-template-system.md](70-template-system.md) | The `domotion template <name>` verb + `Template` contract (DM-1276) — **spike shipped**. Built-ins: `lower-third` (generator), `device-mockup` (decorator), `background-loop` (generator, doc 71). Third-party templates are npm `domotion-template-*` packages. Code in `src/templates/`. The wider first-party library + a Lottie input adapter are filed follow-ups. |
| [71-background-loop-template.md](71-background-loop-template.md) | The `background-loop` built-in template (DM-1280) — **shipped**. A procedural seamlessly-looping animated background: a base fill with N soft `radial-gradient` color blobs that each drift (translate) and breathe (opacity), staggered, infinitely. Deterministic from a `seed`; `aurora` / `orbs` variants. Built on the doc-70 contract + doc-44 repeating animations; uses only origin-safe translate + opacity with `alternate` loops (the two animation constraints are documented). |
| [72-kinetic-text-template.md](72-kinetic-text-template.md) | The `kinetic-text` built-in template (DM-1277) — **shipped**. Kinetic typography: a headline string is expanded at author time into per-word / per-char units, each revealed with a staggered one-shot animation (`rise` / `slide` / `fade`), then held assembled. The clearest showcase of the doc-70 "pre-process once, replay free" thesis. Each unit is a transform-wrapper around an opacity-inner (the same two animation constraints as doc 71). |
| [73-template-frames.md](73-template-frames.md) | **Template frames in the `animate` config** (DM-1287) — **shipped**. A frame may set `"template": "<name>"` (+ `"params"`) instead of `input` / `cast` to embed a named template (doc 70) as a nested animated SVG, composing a `lower-third` / `kinetic-text` / etc. into a larger multi-frame animation. Params validated against the template's schema; template inherits the canvas size (centered otherwise). The crux: template frames are pre-rendered before the outer font lifecycle, and the nested document's global names (ids, font families, `.f-N` classes, `@keyframes`, `--scene-dur`) are namespaced per-frame (`namespaceEmbeddedAnimatedSvg`) so they can't collide with the outer animation or sibling template frames. |
| [74-template-authoring.md](74-template-authoring.md) | **Authoring & publishing a third-party template** (DM-1282) — **shipped**. The hands-on guide for third-party authors: a template is an npm package named `domotion-template-<name>` exporting a `Template` (default / `template` export); the `domotion-svg` peer dep supplies only the types. Covers the package shape, generator vs decorator, the two animation constraints (incl. `transformOrigin`), params/`z.coerce`, `durationMs`, testing via `renderTemplateToSvg`, publishing, and the `domotion-template-*` discovery convention. Runnable scaffold in `examples/template-package/`; public manual page at `site/pages/guides/templates.tsx`. |
